# Теория зависимого типа



## 2.1 Simple Type Theory


Термин «Теория типов» (Type theory) получил свое название из-за того, что каждое выражение имеет связанный с ним тип. Например, в определенном контексте выражение x + 0 может обозначать натуральное число, а f — функцию, заданную на множестве натуральных чисел. Для тех, кто любит точные определения: натуральное число в Lean представляет собой беззнаковое целое число произвольной точности (Это значит, что число не имеет привычные нам лимиты 32, 64 бит. В Lean любое число может быть хоть тысяча знаков и оно не вызовет переполнения, конечно в рамках ОЗУ).Вот несколько примеров того, как можно объявлять объекты в Lean и проверять их типы.

**Примеры объявления объектов и проверки типов**

В Lean вы можете использовать ключевое слово `def` для создания новых объектов, а специальные команды `#check` и `#eval` для проверки их типов и вычисления значений.

```lean
/- Определяем константы. -/

def m : Nat := 1       -- m - это натуральное число
def n : Nat := 0
def b1 : Bool := true  -- b1 - это булево значение
def b2 : Bool := false

/- Проверяем их типы. -/

#check m
#check n
#check n + 0
#check m * (n + 0)
#check b1

-- "&&" - это логическое «И» (конъюнкция)
#check b1 && b2

-- "||" - это логическое «ИЛИ» (дизъюнкция)
#check b1 || b2

-- логическая константа "true" (истина)
#check true

/- Вычисляем значения (запускаем код) -/

#eval 5 * 4
#eval m + 2
#eval b1 && b2
```

Каждая команда, начинающаяся с символа `#`, выводит результат работы в интерактивную панель (Infoview) вашей среды разработки.

Команда `#check` заставляет Lean сообщить тип выражений; в Lean вспомогательные команды, которые запрашивают информацию у системы, обычно начинаются с символа решётки (`#`). Команда `#eval` просит Lean вычислить заданное выражение. 

Вам следует попробовать самостоятельно объявить несколько констант и проверить типы некоторых выражений. Объявление новых объектов таким способом - отличный метод поэкспериментировать с системой.

Любой текст, заключенный между символами /- и -/, представляет собой блок комментариев, который Lean полностью игнорирует. Точно так же два тире -- указывают на то, что вся оставшаяся часть строки является комментарием, который система тоже не учитывает. Блоки комментариев могут быть вложенными - это позволяет временно «комментировать» целые куски кода, как это делается во многих других языках программирования.

```lean
/- 
  Это внешний комментарий, который отключает весь блок ниже

  def a : Nat := 5

  /- Это внутренний (вложенный) комментарий с пояснением к переменной b -/
  def b : Nat := a + 10
-/
```

Что делает простую теорию типов мощной, так это возможность конструировать новые типы на основе уже существующих. 

Например, если `a` и `b` — это типы, то:
* `a → b` (или `a -> b`) обозначает тип функций из `a` в `b`.
* `a × b` обозначает тип пар, состоящих из элемента типа `a` и элемента типа `b` (**декартово произведение**).

Обратите внимание, что `×` и `→` - это символы Юникода. Разумное использование Юникода улучшает читаемость кода, и все современные редакторы отлично его поддерживают. В стандартной библиотеке Lean для обозначения типов часто используются греческие буквы, а символ `→` применяется как более компактная версия стрелки `->`.

```lean
-- 1. Проверка типов пространств функций
#check Nat → Nat
-- Результат: Nat → Nat : Type

#check Nat -> Nat
-- Результат: Nat → Nat : Type

-- 2. Декартово произведение (типы пар)
#check Nat × Nat
-- Результат: Nat × Nat : Type

#check Prod Nat Nat
-- Результат: Nat × Nat : Type

-- 3. Демонстрация правой ассоциативности стрелки функции
#check Nat → Nat → Nat
-- Результат: Nat → Nat → Nat : Type

#check Nat → (Nat → Nat)
-- Результат: Nat → Nat → Nat : Type

-- 4. Изменение приоритета с помощью скобок
#check Nat × Nat → Nat
-- Результат: Nat × Nat → Nat : Type

#check (Nat → Nat) → Nat
-- Результат: (Nat → Nat) → Nat : Type

-- 5. Проверка типов встроенных функций и объектов
#check Nat.succ
-- Результат: Nat.succ (n : Nat) : Nat

#check (0, 1)
-- Результат: (0, 1) : Nat × Nat

#check Nat.add
-- Результат: Nat.add : Nat → Nat → Nat

-- 6. Применение функций и частичное применение (каррирование)
#check Nat.succ 2
-- Результат: Nat.succ 2 : Nat

#check Nat.add 3
-- Результат: Nat.add 3 : Nat → Nat

#check Nat.add 5 2
-- Результат: Nat.add 5 2 : Nat

-- 7. Проекция элементов из пары (кортежа)
#check (5, 9).1
-- Результат: (5, 9).fst : Nat

#check (5, 9).2
-- Результат: (5, 9).snd : Nat

/- Вычисление выражений (запуск калькулятора Lean) -/

#eval Nat.succ 2
-- Результат: 3

#eval Nat.add 5 2
-- Результат: 7

#eval (5, 9).1
-- Результат: 5

#eval (5, 9).2
-- Результат: 9
```

Type - это тип самих типов. В Lean всё структурировано, поэтому даже у Nat или Nat → Nat есть свой тип, который называется Type.

Nat.add 3 - яркий пример каррирования. Так как функция сложения ждет два числа, передав ей только тройку, мы на лету создали новую функцию, которая просто прибавляет 3 к любому числу.

.1 и .2 - операторы, которые «достают» элементы из скобок. Внутри себя Lean переименовывает их в .fst (first) и .snd (second).

Вам стоит еще раз попробовать несколько примеров самостоятельно. 

Давайте рассмотрим базовый синтаксис. Вы можете ввести стрелку Юникода →, набрав \to, \r или \->. Вы также можете использовать альтернативный ASCII-вариант ->, поэтому выражения Nat -> Nat и Nat → Nat означают одно и то же. Оба выражения обозначают тип функций, которые принимают натуральное число на вход и возвращают натуральное число на выходе. Символ Юникода × для декартова произведения вводится как \times. Как правило, для обозначения произвольных типов вы будете использовать строчные греческие буквы, такие как α, β и γ. Конкретно эти буквы можно ввести с помощью команд \a, \b и \g.

Здесь есть еще несколько моментов, на которые стоит обратить внимание. 

Во-первых, применение функции f к значению x записывается просто как f x (например, Nat.succ 2). 

Во-вторых, при написании выражений типов стрелки связываются (ассоциируются) вправо; например, тип функции Nat.add записывается как Nat → Nat → Nat, что эквивалентно Nat → (Nat → Nat). Таким образом, вы можете рассматривать Nat.add как функцию, которая принимает натуральное число и возвращает другую функцию, которая, в свою очередь, принимает натуральное число и возвращает натуральное число. 

В теории типов это, как правило, гораздо удобнее, чем представлять Nat.add как функцию, которая принимает на вход пару натуральных чисел и возвращает натуральное число на выходе. Например, такой подход позволяет вам «частично применять» функцию Nat.add. Приведенный выше пример показывает, что выражение Nat.add 3 имеет тип Nat → Nat, то есть Nat.add 3 возвращает функцию, которая «ждет» второй аргумент n, что в итоге эквивалентно записи Nat.add 3 n.

Вы уже видели, что если у вас есть m : Nat и n : Nat, то запись (m, n) обозначает упорядоченную пару элементов m и n, которая имеет тип Nat × Nat. Это дает вам способ создавать пары натуральных чисел. И наоборот, если у вас есть объект p : Nat × Nat, то вы можете написать p.1 : Nat и p.2 : Nat. Это дает вам способ извлекать два его компонента.


## 2.2 Типы как объекты


Один из способов, с помощью которых теория зависимых типов Lean расширяет простую теорию типов, заключается в том, что сами типы — такие сущности, как `Nat` и `Bool` — являются «гражданами первого класса» (first-class citizens). Это означает, что они сами по себе являются объектами. А раз так, то каждый из них тоже должен иметь свой тип.

> **Важно!** Граждане первого класса (First-class citizens) - в программировании и логике это объекты, с которыми можно обращаться точно так же, как и со всеми остальными: присваивать их переменным, передавать в качестве аргументов в функции и возвращать из функций. В Lean типы данных не просто жестко прописаны в компиляторе, они являются такими же полноценными «живыми» объектами, как и числа.

```lean
#check Nat
-- Ответ Lean: Nat : Type
-- (Тип натуральных чисел сам относится к базовому типу «Type»)

#check Bool
-- Ответ Lean: Bool : Type
-- (Тип булевых значений — это тоже объект типа «Type»)

#check Nat → Bool
-- Ответ Lean: Nat → Bool : Type
-- (Тип функций, принимающих число и возвращающих истину/ложь)

#check Nat × Bool
-- Ответ Lean: Nat × Bool : Type
-- (Тип пар, где первый элемент — число, а второй — булево значение)

#check Nat → Nat
-- Ответ Lean: Nat → Nat : Type

#check Nat × Nat → Nat
-- Ответ Lean: Nat × Nat → Nat : Type

#check Nat → Nat → Nat
-- Ответ Lean: Nat → Nat → Nat : Type

#check Nat → (Nat → Nat)
-- Ответ Lean: Nat → Nat → Nat : Type

#check Nat → Nat → Bool
-- Ответ Lean: Nat → Nat → Bool : Type
-- (Пример типа для бинарных отношений, вроде операции «меньше чем» (x < y))

#check (Nat → Nat) → Nat
-- Ответ Lean: (Nat → Nat) → Nat : Type
```

Вы можете видеть, что каждое из приведенных выше выражений является объектом типа **Type**. Вы также можете объявлять новые константы для самих типов:

```lean
def α : Type := Nat
def β : Type := Bool
def F : Type → Type := List
def G : Type → Type → Type := Prod

#check α
-- Ответ Lean: α : Type
-- (Константа α — это тип, и она равна Nat)

#check F α
-- Ответ Lean: F α : Type
-- (Выражение List Nat — тип списков из натуральных чисел)

#check F Nat
-- Ответ Lean: F Nat : Type
-- (То же самое, только явно указан тип Nat)

#check G α
-- Ответ Lean: G α : Type → Type
-- (Частичное применение: Prod Nat всё еще ждет второй тип, чтобы собрать пару)

#check G α β
-- Ответ Lean: G α β : Type
-- (Тип пар Prod Nat Bool, он же Nat × Bool)

#check G α Nat
-- Ответ Lean: G α Nat : Type
-- (Тип пар Prod Nat Nat, он же Nat × Nat)
```

Как подсказывает приведенный выше пример, вы уже встречали образец функции с типом Type → Type → Type — а именно, декартово произведение `Prod`:

```lean
def α : Type := Nat
def β : Type := Bool

#check Prod α β
-- Ответ Lean: α × β : Type
-- (Lean принимает функцию Prod, но в ответе красиво заменяет её на значок ×)

#check α × β
-- Ответ Lean: α × β : Type
-- (Прямая проверка типа декартова произведения α и β)

#check Prod Nat Nat
-- Ответ Lean: Nat × Nat : Type

#check Nat × Nat
-- Ответ Lean: Nat × Nat : Type
```

Вот еще один пример: для любого заданного типа α, тип List α обозначает тип списков, состоящих из элементов типа α.

```lean
def α : Type := Nat

#check List α
-- Ответ Lean: List α : Type
-- (Поскольку α — это Nat, мы получаем тип списков натуральных чисел)

#check List Nat
-- Ответ Lean: List Nat : Type
-- (Явное указание типа Nat дает абсолютно аналогичный результат)
```

Учитывая, что каждое выражение в Lean имеет тип, вполне естественно задаться вопросом: а какой тип имеет сам `Type` (парадокс Жирара)?

```lean
#check Type
-- Ответ Lean: Type : Type 1
```

Вы только что столкнулись с одним из самых тонких аспектов системы типов Lean. Математический фундамент, лежащий в основе Lean, содержит бесконечную иерархию типов:

```lean
#check Type
-- Ответ Lean: Type : Type 1
-- (Базовая вселенная Type 0 сама является объектом, живущим во вселенной Type 1)

#check Type 1
-- Ответ Lean: Type 1 : Type 2
-- (Вселенная Type 1 является объектом для вселенной более высокого уровня Type 2)

#check Type 2
-- Ответ Lean: Type 2 : Type 3

#check Type 3
-- Ответ Lean: Type 3 : Type 4

#check Type 4
-- Ответ Lean: Type 4 : Type 5
```

Представьте себе `Type 0` как вселенную «малых» или «обычных» типов. В таком случае `Type 1` — это более крупная вселенная типов, которая содержит `Type 0` в качестве своего элемента, а `Type 2` — это еще более масштабная вселенная типов, содержащая в себе `Type 1` как элемент. Этот список бесконечен: вселенная `Type n` существует для каждого натурального числа n. При этом слово `Type` является простым сокращением для `Type 0`:

```lean
#check Type
-- Ответ Lean: Type : Type 1
-- (Проверка базового универсума)

#check Type 0
-- Ответ Lean: Type : Type 1
-- (Результат идентичен, так как Type без цифры по умолчанию означает нулевой уровень)
```

Следующая таблица поможет сделать обсуждаемые взаимосвязи более наглядными и конкретными. Движение вдоль оси X (по горизонтали) представляет собой смену вселенной (уровня), в то время как движение вдоль оси Y (по вертикали) представляет собой изменение того, что иногда называют «степенью» или «порядком» объекта.

sort  Prop (Sort 0)  Type (Sort 1)  Type 1 (Sort 2)  Type 2 (Sort 3)  ...

type  True           Bool           Nat -> Type      Type -> Type 1   ...

term  True.intro     true           fun n => Fin n   fun (_ : Type) => Type   ...

Некоторые операции, однако, должны быть полиморфными относительно вселенных типов (type universes). Например, выражение `List α` должно иметь смысл для абсолютно любого типа `α`, независимо от того, в какой именно вселенной типов живет этот `α`. Это объясняет сигнатуру типа для функции `List`:

```lean
#check List
-- Ответ Lean: List.{u} (α : Type u) : Type u
```

Здесь `u` — это переменная, пробегающая по уровням типов (type levels). Результат выполнения команды `#check` означает, что если α имеет тип `Type u`, то `List α` также имеет тип `Type u`. Функция `Prod` (произведение) аналогично полиморфна:

```lean
#check Prod
-- Ответ Lean: Prod.{u, v} (α : Type u) (β : Type v) : Type (max u v)
```

Чтобы определять полиморфные константы, Lean позволяет вам явно объявлять переменные вселенных с помощью команды `universe`:

```lean
universe u 

def F (α : Type u) : Type u := Prod α α

#check F
-- Ответ Lean: F.{u} (α : Type u) : Type u
```
Вы можете избежать использования команды universe, если укажете параметры вселенной непосредственно при определении F:

```lean
def F.{u} (α : Type u) : Type u := Prod α α

#check F
-- Ответ Lean: F.{u} (α : Type u) : Type u
```


## 2.3 Функция абстракции и оценка
